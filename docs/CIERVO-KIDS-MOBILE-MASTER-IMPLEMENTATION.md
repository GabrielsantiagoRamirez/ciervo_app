**Version:** 1.0  
**Target:** Flutter Mobile App - Ciervo Club  
**Purpose:** Give the mobile AI/developer a complete execution guide to close the full Ciervo Kids ecosystem, including kid login, guardian flows, family chat, wallet, allowed businesses, pay-for-me, location, profile, notifications, UX polish, encoding fixes, and production QA.

---

## 0. Mission

From this point forward, you are responsible for completing the **Ciervo Kids Mobile ecosystem** in Flutter.

Do not treat this as a small feature or isolated screen. Ciervo Kids is a full product area inside Ciervo Club.

Your job is to make it feel complete, safe, simple, modern, and ready for real users.

Do not close the task until:

- The guardian/tutor experience works.
- The kid/minor experience works.
- The app consumes all available backend contracts correctly.
- The app handles pending backend deployment or missing endpoints gracefully.
- There are no debug-looking screens.
- There are no manual IDs exposed to users.
- There are no raw backend enum names visible.
- There are no broken UTF-8 texts such as `reseÃ±as`.
- Chat media shows real images, not URLs.
- Location messages/cards look like a real map experience.
- Share uses the native device share sheet.
- Profile photo upload is truly persistent.
- Kids allowed businesses refresh correctly after saving.
- KYC submit actually creates the request and does not only upload photos.
- `flutter analyze`, tests, and release build pass or all remaining warnings are documented.

Think like Product Owner + UX Designer + Senior Flutter Developer + QA Engineer.

If you find a screen that feels incomplete, debug-like, confusing, or technically exposed, improve it. Do not wait for another prompt.

## 1. Current Backend Context

Backend already analyzed the Kids ecosystem and reported the current situation:

### Existing / reusable backend areas

- Guardian child profile CRUD exists under `/api/guardians/children`.
- Kids wallet exists under guardian-controlled flows.
- Allowed categories and businesses exist under `/api/kids/{kidId}/allowed-*` and business candidates.
- Financial limits and permissions exist.
- Family chat exists under `/api/family/*`, currently mostly guardian/client oriented.
- Payment approval exists partially through `/api/approvals/payment-requests`.
- Generic Pay-for-me exists partially under `/api/payment-requests/pay-for-me`.
- Discovery can be filtered by `kidId` in several queries.

### Previously missing / being implemented by backend

- Role `Kid = 4`.
- Kid login.
- Kid JWT with `accountKind: Kid`, `childProfileId`, `familyId`.
- `/api/kids/me/*` endpoints.
- Kid location share.
- Kid profile photo.
- KidOnly authorization.
- KidAccount table.
- Kid-specific Pay-for-me wrapper.

### Important distinction

There are two different flows:

| Flow | Who logs in | Token |
|---|---|---|
| Guardian using “navigate as this kid” | Adult / guardian | Normal Client token + selected `kidId` in queries |
| Real Kid App experience | Minor / kid | Kid JWT with `role=Kid`, `accountKind=Kid`, `childProfileId` |

Do not mix these flows.

The current guardian mode remains useful for parental controls. The new kid login is a separate limited session.

## 3. Main Product Goal

Ciervo Kids should let a guardian safely manage a child while allowing the child to use a limited app experience.

### Guardian can:

- Create and manage child profiles.
- Create a login account for a child.
- Activate/deactivate child access.
- Change or reset child PIN.
- View child dashboard.
- View child balance.
- Recharge or manage wallet if existing flows support it.
- Approve/reject Pay-for-me requests.
- Configure allowed categories.
- Configure allowed businesses.
- Configure financial permissions and limits.
- View last shared child location.
- Chat with the child/family.
- View child activity/history where backend supports it.

### Kid can:

- Log in with username/PIN.
- See a simple home screen.
- View available balance.
- See allowed businesses only.
- Request “Paga por mí”.
- Share location with family.
- Chat only with family/tutors.
- View wallet movements in read-only mode.
- Change profile photo only.
- Receive notifications.
- Log out.

### Kid cannot:

- Access admin.
- Access business owner tools.
- Access delivery.
- Access settlement account.
- Access adult KYC.
- Search general users.
- Chat with strangers.
- Create businesses.
- See non-allowed businesses.
- Manage adult wallet/cards.
- Modify sensitive profile data.
- See guardian financial accounts.
- Access ads/campaign creation.

Use the app’s existing architecture style. If the project uses Cubit/BLoC, repositories, datasources, DTOs, domain entities, keep the same pattern.

### Required implementation layers

For each new Kid feature, add:

- DTOs / models.
- Remote datasource.
- Repository.
- Cubit/Bloc/controller.
- Page(s).
- Reusable widgets.
- Error states.
- Loading states.
- Empty states.
- Navigation route.
- Tests where feasible.

### Token/session handling

Kid token must not be mixed with guardian token.

Implement one of these safely:

- Separate token storage for Kid and Guardian; or
- One active session at a time with clean logout/switch.

Rules:

- Never persist PIN.
- Never log PIN or token.
- On role `Kid`, restrict navigation to `/kid/*` routes.
- On persistent 401/403, show friendly blocked/session expired state and clear session if needed.
- If guardian switches to kid mode, do not overwrite guardian token with kid token unless the UX explicitly logs out/switches user.
